2. Branch Storekeeper and Supervisor Decision Point

Path A: Item Available in Branch Stock

The branch storekeeper confirms availability and forwards the request to the branch supervisor. After supervisor approval, the branch storekeeper allots the approved item to the employee.

Employee Request→ Branch Storekeeper Checks→ Branch Supervisor Approves→ Branch Storekeeper Allots→ Employee Inventory Updated

Path B: Item Not Available in Branch Stock

The branch storekeeper marks the shortage and forwards to the branch supervisor. If supervisor approves the need, unavailable items move to admin workflow and become required demand for tender planning.

Employee Request→ Branch Storekeeper Marks Shortage→ Branch Supervisor Approves Need→ DD Admin→ AD Admin→ Storekeeper Allots After Approval
Important: Branch storekeeper should record available quantity and shortage quantity. At every approval stage, the approver should be able to change the approved quantity and add comments. The system should preserve requested quantity, available quantity, shortage quantity, approved quantity, actor name, date/time, and comments for audit/history.

3. Group-Wise Admin Workflow

When branch stock is not available, the request is routed according to the group of the requested item. Each group can have its own approval path. A common client-friendly example is:

Branch Supervisor Forward→ DD Admin→ AD Admin-I / AD Admin-II→ Storekeeper→ Allot Item

4. Required Items and Tender Demand

When the branch does not have an item, IMS should not lose that demand. The shortage quantity becomes a required item entry. The tender module receives these pending required items, groups them by item/category, and allows procurement staff to assign them to a new or existing tender.

Branch Shortage Marked→ Required Item Created→ Grouped by Item / Category→ Assigned to Tender→ Procured Stock Allotted

5. Tender Module Assignment Workflow

The tender module should act as the procurement receiver for approved shortages. It should not require procurement staff to re-enter the same requested items manually. Instead, it should pull pending required items from branch/wing requests and assign them to tender records.

Required Items Queue→ Review Demand Summary→ Select Items→ Create / Link Tender→ Tender Items Prepared→ Vendor / PO / Procurement
Tender Step Who Acts What Should Happen System Output
Receive Demand Tender Module Read approved shortages from required items / demand list. Pending demand appears without duplicate manual entry.
Demand Grouping Procurement / Admin Group demands by item, category, branch/wing, urgency, and source request. Consolidated quantity needed for tender planning.
Tender Assignment Procurement / Admin Select required items and assign them to an existing tender or create a new tender. Required item status becomes In Tender and stores tender reference.
Tender Item Creation System / Procurement Create tender item lines from selected demands with quantity and item details. Tender contains demand-backed item quantities.
Procurement Completion Procurement / Storekeeper After vendor, PO, delivery, and stock acquisition, received stock becomes available for issue. Required item can be marked Procured and linked requests can be fulfilled.
Stage Who Acts What They Can Do System Output
Request Submission Employee Select item, enter quantity, write purpose/justification. Request number generated and request status becomes Submitted/Pending.
Availability Check Branch Storekeeper Check branch stock, mark available quantity, mark shortage quantity, write remarks. Request is forwarded to branch supervisor with availability details.
Branch Approval Branch Supervisor Approve, disapprove, change quantity, write comments, or forward shortage to admin workflow. Available items go to branch storekeeper for allotment; unavailable items go to admin workflow.
Admin Group Workflow DD Admin / AD Admin Review item group, change approved quantity, approve/reject/forward, write comments. Approval history is updated and request moves to next admin step.
Demand Capture System / Procurement Record unavailable requested items as pending required items. Tender module receives the demand list for tender assignment.
Tender Assignment Procurement / Tender Module Group required items, select demand rows, and assign them to a new or existing tender. Required items are linked to tender reference and tender items can be prepared from real demand.
Allotment / Issuance Storekeeper Verify final approved item and quantity, allot stock to employee. Stock is deducted from store inventory and added to employee inventory/issued items.
Completion System Record final issued quantity, issuer, issue date, and inventory ledger movement. Request status becomes Issued/Completed and appears in history/reports.

Store Inventory

Deduct: Final issued quantity is subtracted from the issuing storekeeper's stock.

Example: 20 available - 3 issued = 17 remaining.

→

→

Employee Inventory

The requester can later see it in My Inventory / Issued Items.

7. End-to-End Example

  1. Employee of a branch requests 5 laptops.
  2. Branch storekeeper receives the request first and checks local branch stock.
  3. If 3 laptops are available, branch storekeeper records 3 available and 2 short, then forwards to the branch supervisor.
  4. Branch supervisor approves or disapproves the request and can adjust approved quantity with comments.
  5. For the available 3 laptops, branch storekeeper allots them to the employee after supervisor approval.
  6. For the 2 short laptops, IMS records required demand and forwards the shortage to admin workflow based on the laptop item group.
  7. The tender module receives those 2 laptops as pending required demand, groups them with similar requests, and assigns them to a new or existing tender.
  8. DD Admin and AD Admin review the shortage request, can adjust quantity, and record comments.
  9. After tender, PO, delivery, and stock acquisition, storekeeper allots the approved/procured quantity to the employee.
  10. IMS deducts the issued quantity from store inventory, writes stock transaction history, updates employee inventory, and keeps the shortage demand available for tender planning until fulfilled.
