Overview

This document outlines the hybrid approval workflow system that combines employee hierarchy-based approvals (Phase 1) with fixed workflow chain approvals (Phase 2).

Key Concept: Individual requests flow through the employee's management chain first, then proceed to the administrative workflow. Department requests skip the hierarchy and go directly to the administrative workflow.

Database Structure Requirements

Required Table Modifications

Table New Column Data Type Description
request_approvals approval_phase VARCHAR(20) Values: 'HIERARCHY' or 'WORKFLOW'
request_approvals hierarchy_level INT Tracks level in hierarchy (1 = immediate boss, 2 = boss's boss, etc.)
request_approvals workflow_step INT Tracks step in workflow (1 = DD Admin, 2 = Direct Admin, 3 = Chief)
request_approvals request_source VARCHAR(20) Values: 'INDIVIDUAL' or 'DEPARTMENT'

New Table: approval_workflow_steps

Column Data Type Description
id UNIQUEIDENTIFIER Primary Key
workflow_id UNIQUEIDENTIFIER References approval_workflows
step_number INT 1, 2, 3 (DD Admin, Direct Admin, Chief)
role_name VARCHAR(100) DD Admin, Direct Admin, Chief
user_id UNIQUEIDENTIFIER References vw_users_hierarcy.Id
is_active BIT Enable/disable step

Key Decision Points

Questions to Confirm

Question Options/Considerations
1. Hierarchy Depth Limit • Should there be a maximum number of hierarchy levels?
• Or continue until BossID is NULL (top of organization)?
• Consider: Some organizations have 5-10+ levels
2. Workflow Step Configuration • Confirm sequence: DD Admin → Direct Admin → Chief
• Are these three steps always required?
• Can any step be skipped based on request type?
• Should step assignment be dynamic or fixed per role?
3. Request Source Identification • How to identify INDIVIDUAL vs DEPARTMENT request?
• Option A: User selects at submission time
• Option B: Based on requester's role/position
• Option C: Based on item quantity or total value
4. Rejection Handling • If rejected in Phase 1 (hierarchy), request ends immediately?
• If rejected in Phase 2 (workflow), request ends immediately?
• Should rejected requests be resubmittable?
• Notification to original requester on rejection?
5. Forwarding Capability • Can hierarchy-level approvers forward to peers?
• Can workflow approvers forward?
• If forwarded, does it stay at same level or skip?
• Track forwarding chain in approval_history?

Implementation Summary

Phase 1: Hierarchy Approval

  • Query vw_users_hierarcy
  • Start with requester's BossID
  • Create approval record with phase='HIERARCHY'
  • On approval, check next BossID
  • Loop until BossID is NULL
  • Transition to Phase 2

Phase 2: Workflow Approval

  • Query approval_workflow_steps
  • Start with step_number = 1 (DD Admin)
  • Create approval record with phase='WORKFLOW'
  • On approval, increment workflow_step
  • Continue through steps 2 and 3
  • Final approval → request ready
Next Steps: Once business rules are confirmed, the backend approval logic in backend-server.cjs (lines 1010-1050) will be updated to implement this two-phase hybrid system.
